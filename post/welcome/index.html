Обзор Spring Framework Spring упрощает создание корпоративных приложений Java. Он предоставляет всё необходимое для внедрения языка Java в корпоративную среду, включая поддержку Groovy и Kotlin в качестве альтернативных языков на JVM, а также гибкость для создания различных архитектур в зависимости от потребностей приложения. Начиная с версии Spring Framework 6.0, Spring требует Java 17+. Spring поддерживает широкий спектр сценариев использования. В крупных предприятиях приложения часто существуют долгое время и должны работать на JDK и сервере приложений, цикл обновления которых не контролируется разработчиком. Другие приложения могут работать как единый jar-файл со встроенным сервером, возможно, в облачной среде. Третьи могут представлять собой автономные приложения (например, пакетные или интеграционные), которым не нужен сервер. Spring — платформа с открытым исходным кодом. У неё большое и активное сообщество, которое постоянно предоставляет обратную связь, основанную на разнообразных реальных примерах использования. Это способствовало долгому успешному развитию Spring. Что мы подразумеваем под «Spring» Термин «Spring» имеет разное значение в разных контекстах. Он может использоваться для обозначения самого проекта Spring Framework, с которого всё началось. Со временем на основе Spring Framework были созданы другие проекты Spring. Чаще всего, когда говорят «Spring», имеют в виду всё семейство проектов. В этой справочной документации основное внимание уделяется основе: самому Spring Framework. Spring Framework разделён на модули. Приложения могут выбирать необходимые им модули. В основе лежат модули основного контейнера, включая модель конфигурации и механизм внедрения зависимостей. Помимо этого, Spring Framework обеспечивает фундаментальную поддержку различных архитектур приложений, включая обмен сообщениями, транзакционные данные и персистентность, а также веб-приложения. Он также включает в себя веб-фреймворк Spring MVC на основе сервлетов и, параллельно, реактивный веб-фреймворк Spring WebFlux. Замечание о модулях: jar-файлы Spring Framework позволяют выполнять развертывание в модульном каталоге (Java Module System). Для использования в модульных приложениях jar-файлы Spring Framework содержат записи манифеста Automatic-Module-Name, которые определяют стабильные имена модулей на уровне языка (spring.core, spring.context и т. д.), независимые от имён jar-артефактов. Эти jar-файлы следуют тому же шаблону именования, используя «-» вместо «.», например, spring-core и spring-context. Конечно, jar-файлы Spring Framework также отлично работают в classpath. История Spring и фреймворка Spring Spring появился в 2003 году как ответ на сложность ранних спецификаций J2EE. Хотя некоторые считают Java EE и её современную версию Jakarta EE конкурентами Spring, на самом деле они дополняют друг друга. Модель программирования Spring не охватывает спецификацию платформы Jakarta EE; вместо этого она интегрируется с тщательно отобранными отдельными спецификациями из традиционной среды EE: Servlet API (JSR 340) WebSocket API (JSR 356) Concurrency Utilities (JSR 236) JSON Binding API (JSR 367) Bean Validation (JSR 303) JPA (JSR 338) JMS (JSR 914) а также настройки JTA/JCA для координации транзакций, если это необходимо. Spring Framework также поддерживает спецификации Dependency Injection (JSR 330) и Common Annotations (JSR 250), которые разработчики приложений могут использовать вместо специфичных для Spring механизмов, предоставляемых Spring Framework. Изначально они были основаны на стандартных пакетах Javax. Начиная с версии Spring Framework 6.0, Spring был обновлён до уровня Jakarta EE 9 (например, Servlet 5.0+, JPA 3.0+) на основе пространства имён Jakarta вместо традиционных пакетов Javax. С EE 9 как минимальной версией и уже поддерживаемой EE 10, Spring готов обеспечить готовую поддержку для дальнейшего развития API Jakarta EE. Spring Framework 6.0 полностью совместим с Tomcat 10.1, Jetty 11 и Undertow 2.3 в качестве веб-серверов, а также с Hibernate ORM 6.1. Со временем роль Java/Jakarta EE в разработке приложений претерпела изменения. На заре развития J2EE и Spring приложения создавались для развертывания на сервере приложений. Сегодня, благодаря Spring Boot, приложения создаются с учётом требований DevOps и облачных технологий, со встроенным контейнером сервлетов и простой в настройке. Начиная с версии Spring Framework 5, приложение WebFlux даже не использует API сервлетов напрямую и может работать на серверах (например, Netty), которые не являются контейнерами сервлетов. Spring продолжает внедрять инновации и развиваться. Помимо Spring Framework, существуют и другие проекты, такие как Spring Boot, Spring Security, Spring Data, Spring Cloud, Spring Batch и другие. Важно помнить, что у каждого проекта есть свой репозиторий исходного кода, система отслеживания ошибок и периодичность релизов. Полный список проектов Spring см. на сайте spring.io/projects. Философия проектирования Изучая фреймворк, важно понимать не только его предназначение, но и принципы, которым он следует. Вот руководящие принципы Spring Framework: Предоставьте выбор на каждом уровне. Spring позволяет откладывать принятие решений по проектированию как можно позже. Например, вы можете переключать поставщиков персистентности через конфигурацию, не изменяя код. То же самое относится ко многим другим инфраструктурным вопросам и интеграции со сторонними API. Учитывайте различные точки зрения. Spring ценит гибкость и не навязывает свои решения. Он поддерживает широкий спектр потребностей приложений с различными точками зрения. Поддерживайте надежную обратную совместимость. Развитие Spring тщательно контролировалось, чтобы минимизировать критические изменения между версиями. Spring поддерживает тщательно подобранный диапазон версий JDK и сторонних библиотек для упрощения обслуживания приложений и библиотек, зависящих от Spring. Заботьтесь о дизайне API. Команда Spring уделяет много внимания и времени созданию интуитивно понятных API, которые сохранят актуальность в течение многих лет и версий. Устанавливайте высокие стандарты качества кода. Spring Framework уделяет особое внимание содержательному, актуальному и точному документированию Javadoc. Это один из немногих проектов, который может похвастаться чистой структурой кода без циклических зависимостей между пакетами.
