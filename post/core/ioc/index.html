Введение в Spring IoC-контейнер и компоненты Bean В этой главе рассматривается реализация принципа инверсии управления (IoC) в Spring Framework. Внедрение зависимостей (DI) — это специализированная форма IoC, при которой объекты определяют свои зависимости (то есть другие объекты, с которыми они работают) только через аргументы конструктора, аргументы фабричного метода или свойства, устанавливаемые для экземпляра объекта после его создания или возврата из фабричного метода. Затем IoC-контейнер внедряет эти зависимости при создании компонента Bean. Этот процесс, по сути, является обратным (отсюда и название — Инверсия управления) по отношению к самому компоненту Bean, управляющему созданием или расположением его зависимостей с помощью прямого конструирования классов или такого механизма, как шаблон Service Locator. Пакеты org.springframework.beans и org.springframework.context составляют основу IoC-контейнера Spring Framework. Интерфейс BeanFactory предоставляет расширенный механизм конфигурации, способный управлять любым типом объектов. ApplicationContext — это подинтерфейс BeanFactory. Он добавляет: Упрощенную интеграцию с функциями АОП Spring Обработку ресурсов сообщений (для использования при интернационализации) Публикацию событий Контексты, специфичные для уровня приложения, такие как WebApplicationContext, для использования в веб-приложениях. Короче говоря, BeanFactory предоставляет структуру конфигурации и базовую функциональность, а ApplicationContext добавляет функциональность, специфичную для корпоративного использования. ApplicationContext — это полное надмножество BeanFactory, которое используется исключительно в этой главе при описании контейнера IoC Spring. Подробнее об использовании BeanFactory вместо ApplicationContext см. в разделе, посвященном API BeanFactory. В Spring объекты, составляющие основу вашего приложения и управляемые контейнером IoC Spring, называются компонентами (bean-компонентами). Компонент (bean-компонент) — это объект, экземпляр которого создается, собирается и управляется контейнером IoC Spring. В противном случае компонент (bean-компонент) — это просто один из множества объектов в вашем приложении. Компоненты и зависимости между ними отражаются в метаданных конфигурации, используемых контейнером. 1,547 / 5,000 Обзор контейнера Интерфейс org.springframework.context.ApplicationContext представляет контейнер Spring IoC и отвечает за создание, настройку и сборку компонентов. Контейнер получает инструкции по созданию, настройке и сборке компонентов, считывая метаданные конфигурации. Метаданные конфигурации могут быть представлены в виде аннотированных классов компонентов, классов конфигурации с фабричными методами, внешних XML-файлов или скриптов Groovy. Используя любой из этих форматов, вы можете создавать свои приложения и создавать обширные взаимозависимости между этими компонентами. Несколько реализаций интерфейса ApplicationContext являются частью ядра Spring. В автономных приложениях обычно создается экземпляр AnnotationConfigApplicationContext или ClassPathXmlApplicationContext. В большинстве сценариев применения для создания одного или нескольких экземпляров контейнера Spring IoC не требуется явный пользовательский код. Например, в сценарии простого веб-приложения достаточно простого шаблонного XML-дескриптора веб-страницы в файле web.xml приложения (см. раздел Удобное создание экземпляра ApplicationContext для веб-приложений). Классы вашего приложения объединены с метаданными конфигурации, поэтому после создания и инициализации ApplicationContext вы получаете полностью настроенную и исполняемую систему или приложение.
